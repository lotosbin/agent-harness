# 第 2 篇：核心工作流详解

> AI 编程工作流 | TDD 教程 | AI 代码审查

## 7 步工作流全景

```
想法 → 头脑风暴 → 设计评审 → 隔离工作区 → 任务规划 → 执行开发 → 代码审查 → 合并发布
```

每一步都有明确的输入、输出和验证标准，确保 AI 的每一步操作都可追踪、可审查。

---

## Step 1：头脑风暴（Brainstorming）

**做什么**：AI 通过苏格拉底式提问帮你完善想法，直到你认可设计方案。

### 关键点

- AI 会先了解项目背景（读取文件、分析代码）
- 一次只问一个问题
- 设计分章节展示，你需要逐节确认
- 全部确认后生成设计文档并提交到 git

### 示例对话

```
你：我想给博客加个评论系统
AI：我正在使用 brainstorming 技能...
    你的项目是 Web 应用、CLI 工具还是 API 服务？

你：FastAPI 写的博客后端 API
AI：评论需要支持嵌套回复吗？

你：需要，最多三层
AI：回复是实时显示还是需要审核？

你：需要审核，管理员可以开关这个功能
AI：好的，我已经了解了需求。让我整理一下设计方案...
```

### 适合场景

- 新功能开发
- 架构调整
- 技术选型决策
- Bug 修复方案讨论

---

## Step 2：创建隔离工作区（Git Worktree）

**做什么**：为新功能创建独立的 git worktree，避免污染主分支。

### 什么是 Git Worktree？

Git Worktree 允许你在同一仓库的不同目录中同时工作，每个目录对应不同的分支。

### 好处

| 好处 | 说明 |
|------|------|
| **完全隔离** | 每个功能在独立目录，不影响其他工作 |
| **并行开发** | 同时开发多个功能，互不干扰 |
| **安全可靠** | 出错不影响主分支，随时可丢弃 |
| **快速切换** | 不需要 git stash，随时切换上下文 |

### 使用方式

AI 会自动调用 `using-git-worktrees` 技能，创建类似这样的隔离环境：

```
your-project/
├── .worktrees/
│   ├── feature/blog-comments/    # 评论功能开发中
│   └── feature/user-auth/       # 用户认证开发中
└── main/                        # 主分支
```

## Step 3：编写实施计划（Writing Plans）

**做什么**：将任务分解为 2-5 分钟的微小步骤，每个步骤包含：

- 精确的文件路径
- 完整的代码示例
- 验证命令和预期输出

### 关键原则：2-5 分钟原则

每个任务步骤应该足够小，能在 2-5 分钟内完成并验证。

**好处**：
- 错误容易定位（步骤短）
- 审查快速（代码量少）
- 进度可见（每步都有产出）
- 心理负担低（不会拖延）

### 计划示例

```markdown
### Task 1: 创建评论数据模型

**文件：**
- 创建: `models/comment.py`
- 测试: `tests/models/test_comment.py`

- [ ] **Step 1: 编写失败的测试**
- [ ] **Step 2: 运行测试，确认失败**
- [ ] **Step 3: 编写最小实现**
- [ ] **Step 4: 运行测试，确认通过**
- [ ] **Step 5: 提交代码**
```

---

## Step 4：Subagent 驱动开发

**做什么**：AI 为每个任务启动一个独立的子 Agent，完成后进行两阶段审查。

### 工作原理

```
主 Agent：分发 Task 1 给子 Agent
         ↓
子 Agent：编写测试 → 实现代码 → 自测
         ↓
主 Agent：审查子 Agent 的代码
         ↓
主 Agent：分发 Task 2 给子 Agent
         ↓
...
```

### 优势

| 优势 | 说明 |
|------|------|
| **上下文干净** | 每个任务都是全新开始，不受历史污染 |
| **质量可控** | 两阶段审查（Spec 审查 + 代码质量审查） |
| **快速迭代** | 子 Agent 并行处理多个任务 |
| **可追溯** | 每个任务独立提交，历史清晰 |

## Step 5：测试驱动开发（TDD）

**做什么**：严格遵循红-绿-重构循环。

### 红-绿-重构循环

```
┌─────────────────┐
│  RED（红）      │
│  写失败的测试    │
└────────┬────────┘
         ↓
┌─────────────────┐
│  验证失败        │
│  确认是预期失败  │
└────────┬────────┘
         ↓
┌─────────────────┐
│  GREEN（绿）    │
│  写最小代码      │
└────────┬────────┘
         ↓
┌─────────────────┐
│  验证通过        │
│  所有测试绿      │
└────────┬────────┘
         ↓
┌─────────────────┐
│  REFACTOR      │
│  重构优化        │
└─────────────────┘
```

### 铁律

> **没有先失败的测试，就不能写生产代码。**

### 代码示例

```python
# RED：写测试
def test_user_login():
    """正确密码应该登录成功，错误密码应该返回错误"""
    result = login("admin", "wrong")
    assert result.error == "Invalid credentials"
```

```bash
# 运行测试，确认失败
$ pytest test_user_login.py
FAIL: expected 'Invalid credentials', got None
```

```python
# GREEN：最小实现（只让测试通过，不多不少）
def login(username, password):
    if password != "correct_password":
        return {"error": "Invalid credentials"}
    return {"success": True}
```

```bash
# 运行测试，确认通过
$ pytest test_user_login.py
PASS
```

```python
# REFACTOR：改善代码（可选）
def login(username, password):
    if not verify_password(username, password):
        return {"error": "Invalid credentials"}
    return {"success": True}
```

## Step 6：代码审查（Requesting Code Review）

**做什么**：对照实施计划逐项检查，发现问题按严重程度分类报告。

### 审查报告格式

```markdown
## 代码审查报告 - Task 1: 评论数据模型

### 严重问题（必须修复）
- ❌ 未处理空用户名的边界情况
- ❌ 嵌套层级计算有误

### 建议改进（可选）
- 💡 可以将错误消息提取为常量
- 💡 添加类型注解提高可读性

### 通过项
- ✅ 测试覆盖率符合要求
- ✅ 遵循项目代码风格
- ✅ 数据模型设计合理
```

### 严重程度分级

| 级别 | 标识 | 含义 |
|------|------|------|
| 严重 | ❌ | 必须修复，否则功能不可用或存在安全风险 |
| 建议 | 💡 | 建议修复，提升代码质量 |
| 通过 | ✅ | 已验证，符合要求 |

---

## Step 7：完成开发分支

**做什么**：最终验证，准备合并。

### 最终检查清单

- ✅ 所有测试通过
- ✅ 代码审查问题已修复
- ✅ 文档已更新
- ✅ 没有 lint 错误

### 合并选项

| 选项 | 说明 | 适用场景 |
|------|------|----------|
| **Squash Merge** | 将所有提交压缩为一个 | 保持主分支干净 |
| **Merge Commit** | 保留完整提交历史 | 需要完整历史追溯 |
| **Rebase + Merge** | 先变基再合并 | 最线性的历史 |

## 为什么"先写测试"这么重要？

### 常见误区

| 误区 | 真相 |
|------|------|
| "代码简单，不需要测试" | 简单代码也会 Bug，测试只需 30 秒 |
| "我手动测试过了" | 手动测试无法复现，不能防止回归 |
| "先写代码再补测试也一样" | 测试通过只能证明你记得测什么，无法发现遗漏 |

### TDD 的价值

1. **强制思考**：写测试前必须想清楚"期望什么行为"
2. **即时反馈**：每步都验证，出问题立即知道
3. **防止退化**：后续改动不会破坏已有功能
4. **活文档**：测试就是代码行为的说明书

---

## 为什么流程比 AI 聪明更重要？

AI 很聪明，但：

- **不可靠**：同一问题可能给出不同答案
- **健忘**：每次对话都是新的上下文
- **无纪律**：倾向于快速"搞定"而非正确做事

Superpowers 的价值不是让 AI 更聪明，而是**让 AI 的行为可预测、可追踪**。
