## 第一层：问题现象

### 你的 AI 编程是否这样？

| 问题 | 场景 | 频率 |

## 第二层：表面原因

### 人们通常以为是 AI 不够聪明

```
❌ 错觉：GPT-5 不够强，换个模型就好了
❌ 错觉：Claude 不够好，试试 Claude 4
❌ 错觉：Cursor 更好用，换 IDE
```

**但换了无数工具，问题依然存在。**

---

## 第三层：深层原因

### 真正的问题是：AI 缺乏稳定的流程

| 表面原因 | 深层原因 |
|----------|----------|
| 代码跑不起来 | AI 跳过了测试，直接写实现 |
| Bug 越修越多 | 没有系统化的调试流程 |
| 上下文丢失 | 没有持久化的设计文档 |
| 代码看不懂 | 没有先设计再实现 |
| 重复造轮子 | 没有任务规划，想到哪写到哪 |

**核心问题不是 AI 能力，而是 AI 行为没有约束。**

---

## 第四层：根本原因

### AI 的三大缺陷

```
┌─────────────────────────────────────────────────────────┐
│                    AI 的三大缺陷                         │
├─────────────────────────────────────────────────────────┤
│  1. 不可靠：同一问题可能给出不同答案                      │
│  2. 健忘症：每次对话都是新的上下文                        │
│  3. 无纪律：倾向于快速"搞定"而非正确做事                  │
└─────────────────────────────────────────────────────────┘
```

**这三点决定了：没有流程约束的 AI 编程，就是一场赌博。**

## 第五层：解决方案

### Superpowers 的因果解法

```
AI 缺陷              →  Superpowers 解法              →  效果
────────────────────────────────────────────────────────────
不可靠               →  强制 TDD，先写测试             →  每步验证
健忘症               →  设计文档持久化                 →  上下文延续
无纪律               →  流程约束，7 步工作流           →  行为可预测
```

### Superpowers 的 7 步工作流

```
1. 头脑风暴     → 约束：必须先设计，不能直接写代码
2. Git Worktree → 约束：必须隔离环境，不污染主分支
3. 任务规划     → 约束：必须拆分任务，不能一口吃
4. Subagent 开发 → 约束：必须独立执行，不能混在一起
5. TDD 测试     → 约束：必须先写测试，不能跳过
6. 代码审查     → 约束：必须检查问题，不能盲目自信
7. 完成分支     → 约束：必须验证通过，不能草率合并
```

**每一步都有明确的输入、输出、验证标准。AI 的行为被流程约束。**

## 第六层：实际效果

### 效果对比

| 指标 | 无流程 AI | Superpowers |
|------|----------|-------------|
| 代码质量 | 靠运气 | 流程保证 |
| Bug 率 | 高（3-5 个/功能） | 低（0-1 个/功能） |
| 上下文管理 | 每次丢失 | 持久化保存 |
| 团队协作 | 难以交接 | 文档可传承 |
| 调试效率 | 1 小时+/Bug | 10 分钟/Bug |

### 真实案例

```
无流程 AI：
"帮我加个登录功能"
→ AI 生成 500 行代码
→ 运行报错
→ 让 AI 修
→ 引入新 Bug
→ 继续修...
→ 2 小时后：代码自己都看不懂

Superpowers：
"帮我加个登录功能"
→ 头脑风暴：明确需求
→ 设计文档：确定方案
→ 任务规划：拆分为 5 个小任务
→ TDD 开发：每步测试通过
→ 代码审查：发现并修复问题
→ 1.5 小时后：功能完整，测试覆盖，代码可维护
```
