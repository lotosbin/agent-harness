## 二、Terminal-Bench 基准评测

### 2.1 评测概述

Terminal-Bench 是 Factory AI 开源的**终端任务完成率基准**，衡量 AI Agent 在真实终端环境中完成端到端复杂任务的能力。

**核心数据**:
- 任务数量：80 个经人工验证的容器化任务
- 运行策略：每个模型 5 轮独立运行
- 环境：Ubuntu 非交互任务模式，沙箱隔离
- 评测指标：Task Resolution Rate（所有后置测试通过即视为成功）

### 2.2 任务分类（6 大领域）

| 类别 | 示例任务 |
|------|----------|
| 编码 (Coding) | 多语言代码生成、重构 |
| 构建/测试 (Build/Test) | 依赖管理、CI 流水线配置 |
| 数据与 ML 工作流 | RL 训练、文本分类器训练 |
| 系统与网络 | Git Web 服务器配置、网络诊断 |
| 安全 (Security) | 仓库密钥清理、安全漏洞扫描 |
| CLI 工作流 | 复杂终端命令序列执行 |

### 2.3 评测结果

| 排名 | Agent | 模型 | 得分 |
|------|-------|------|------|
| 🥇 1 | **Droid** | Opus 4.1 | **58.8%** |
| 2 | **Droid** | GPT-5 | 52.5% |
| 3 | **Droid** | Sonnet 4 | 50.5% |
| 4 | Claude Code | Opus | 43.2% |
| 5 | Codex CLI | — | 42.8% |

> **关键洞察**: Droid with Sonnet 4（50.5%）超越了所有其他基于 Opus 的 Agent（43.2%）。Factory 官方结论：**"Agent 设计比模型选择更具决定性"**。

## 三、核心架构设计

### 3.1 三层提示工程体系

```
┌─────────────────────────────────────────────┐
│  Layer 3: System Notifications（系统通知）   │  ← 环境事件触发
├─────────────────────────────────────────────┤
│  Layer 2: System Prompts（系统提示）         │  ← 角色定义、行为规范
├─────────────────────────────────────────────┤
│  Layer 1: Tool Descriptions（工具描述）      │  ← 最小化工具集设计
└─────────────────────────────────────────────┘
```

### 3.2 核心设计原则

1. **模型专属架构**: 不同 LLM 提供商使用不同内部架构适配
2. **最小化工具设计**: 减少工具数量以降低错误率
3. **系统与环境感知引导**: 启动时自动感知上下文
4. **速度优化**: 高效工具实现，减少 LLM 等待时间

### 3.3 未来技术方向

- **多 Agent 架构**: 并行任务执行
- **高级记忆系统**: 持续学习能力
- **全场景自动化**: 覆盖 CLI、IDE、CI/CD、Server 全链路

## 四、产品生态

### 4.1 产品矩阵

```
Factory AI ─┬─ Droid CLI          # 核心终端 Agent
           ├─ Droid Action        # GitHub Action（PR 审查、安全扫描）
           ├─ IDE 插件            # VS Code / JetBrains / Zed
           ├─ droid-sdk-python    # Python SDK（asyncio + JSON-RPC）
           ├─ droid-sdk-typescript# TypeScript SDK（流式 + 会话管理）
           └─ factory-plugins     # 插件市场
```

### 4.2 SDK 技术规格

#### Python SDK (`droid-sdk`)

| 项目 | 规格 |
|------|------|
| 最低版本 | Python 3.10+ |
| 传输协议 | JSON-RPC 2.0 over subprocess stdin/stdout |
| 核心类 | `DroidClient` + `ProcessTransport` |
| 事件类型 | `AssistantTextDelta`, `ThinkingTextDelta`, `ToolUse`, `ToolResult`, `ToolProgress`, `TurnComplete` 等 |
| 错误类型 | `ConnectionError`, `ProtocolError`, `SessionError`, `TimeoutError` |
| 开发工具 | `uv` + pytest + mypy(strict) + ruff |

#### TypeScript SDK (`@factory/droid-sdk`)

| 项目 | 规格 |
|------|------|
| 最低版本 | Node.js 18+ |
| 传输协议 | JSON-RPC 2.0 over subprocess |
| 核心接口 | `query()`, `createSession()`, `resumeSession()` |
| 流式支持 | AsyncGenerator 逐 token 流式输出 |
| MCP 支持 | `addMcpServer()`, `removeMcpServer()` |
| 安装 | `npm install @factory/droid-sdk` |

### 4.3 Droid Action 功能

| 命令 | 功能描述 |
|------|----------|
| `@droid fill` | 生成标准化 PR 描述 |
| `@droid review` | 内联代码审查（支持 deep/shallow 两种深度） |
| `@droid security` | STRIDE 安全审查 |
| `@droid security --full` | 全仓库安全扫描 |

**Review 深度预设**:
- `deep`: 使用 `gpt-5.2` + 高推理投入
- `shallow`: 使用 `kimi-k2-0711` + 默认推理

## 七、关键结论

1. **Agent 设计 > 模型选择**: Droid + Sonnet 4 超越所有 Opus 模型，证明框架优化价值大于模型切换
2. **SDK 生态完整**: 唯一同时提供 Python/TypeScript SDK 的终端 Agent，支持 MCP 协议
3. **全流程覆盖**: 从 PR 审查到终端任务，覆盖开发全生命周期
4. **Benchmark 开源**: Terminal-Bench 对外开放，推动行业评测标准化
